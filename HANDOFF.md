## 1. What IntelliX is

A multi-language (Python/C++/Java) ML-powered code performance profiler. Solo portfolio project (explicitly decided — see §3). Flask REST API + scikit-learn classifiers (RF/GB/SVM/MLP) trained on synthetic template-generated code + SQLite history + Docker/CI + an MCP server wrapper. Core distinction: **Python uses real AST analysis** (the `ast` module); **C++/Java have always used a text-pattern heuristic** (counting `for`/`while`/brace occurrences as raw strings) — this is being actively upgraded right now (§6/§7).

## 5. Key architectural facts to hold in your head

- `profiler_main.py`'s `CodeProfiler.profile_cpp()`/`profile_java()` are the methods that currently do the OLD text-heuristic counting. These are what need to be rewired to call the new analyzers.
- `FEATURE_COLUMNS` (in `profiler_main.py`) currently = `['execution_time_ms', 'memory_usage_kb', 'loop_depth', 'max_nesting_depth', 'function_calls', 'conditionals', 'complexity_score', 'language']`. Adding a `stdlib_complexity_weight` feature (see §6) means this list changes, which means: `generate_dataset()`'s output columns change, the model needs **retraining** (the feature space changed), and `MLPipeline.predict()`'s feature-vector construction needs the new field added with a safe default for backward compatibility if an old saved model is loaded against new code (or just retrain and don't worry about old artifacts — old `model_*.joblib` files aren't meant to be portable across feature-set changes anyway).
- `_compute_features(code, lang)` is the single function both training (`generate_dataset`) and... actually, no — **important nuance**: `_compute_features` is only used during *dataset generation*, not during live inference. Live inference goes through `CodeProfiler.analyze()` → `CodeProfiler.profile()` → `profile_python`/`profile_cpp`/`profile_java()` directly. Both paths call into the same `profile_*` methods underneath (via `CodeProfiler().profile(code, lang)` inside `_compute_features`), so changing `profile_cpp`/`profile_java`'s internals automatically affects both training-data generation and live inference consistently — **this consistency is critical and must be preserved**, it's exactly what avoids the train/serve skew problem that was explicitly reasoned through and avoided earlier in this project (see §6 rationale).
- The benchmark suites (`benchmarks/benchmark_validation.py` and `benchmarks/real_templates/real_template_validation.py`) are the **regression tests** for this whole effort. After integrating the new analyzers, **both must be re-run** to prove the specific findings they documented (especially the `std::reverse`/`std::count`/`stream`/`HashSet` blind spots and the bubble-sort mislabel) actually improve. Don't just trust that they improve — that would break the rigor standard this entire project has held throughout.

## 6. What was just built (verified working, not yet wired in)

**`cpp_analyzer.py`** — `CppTreeSitterAnalyzer`, using `tree-sitter` + `tree_sitter_cpp`. Two upgrades over the old heuristic, both zero-execution-risk (pure static parsing):
1. Real parsing (concrete syntax tree) instead of substring matching — fixes correctness issues like keywords inside strings/comments, and correctly distinguishes `for_statement` from C++'s separate `for_range_loop` node type (range-based for loops — **this was a real bug caught and fixed mid-build**: the first version of the analyzer reported `loops=0` for `for (int x : arr)` because that's a distinct grammar node, not `for_statement`).
2. A `KNOWN_COMPLEXITY` lookup dict mapping recognized `std::` algorithm names (`sort`→1.5, `reverse`/`count`/`find`/`accumulate`/etc.→1.0, `lower_bound`/`upper_bound`→0.3) to a "loop-equivalent weight," extracted via `call_expression`'s **named field** `function` (robust — confirmed via `node.child_by_field_name('function')`/`('arguments')`, not fragile positional child indexing) and a helper that finds the rightmost `identifier`/`field_identifier` in that subtree (correctly distinguishes `std::reverse` → "reverse" from `r.begin()` → "begin").

Verified against the exact cases the real-template audit found broken:
```
string_reverse: loops=0 stdlib_weight=1.0   (was invisible to old heuristic — FIXED)
std_count:      loops=0 stdlib_weight=1.0   (was invisible — FIXED)
single_pass_sum: loops=1 (after for_range_loop fix)   (control, correct)
bubble_sort:    loops=2 nesting=2            (control, correct, unchanged from before)
naive_fib:      loops=0                      (control, correct — recursive, no loop)
prime_sieve:    loops=2 nesting=2            (correctly locates the overflow-bug-adjacent loop structure)
```

**`java_analyzer.py`** — `JavaTreeSitterAnalyzer`, using `tree-sitter` + `tree_sitter_java`. Same two upgrades:
1. Real parsing — fixes the same class of bug: Java's for-each loop is a distinct `enhanced_for_statement` node, not `for_statement` (**same bug pattern as C++, caught and fixed the same way**).
2. `KNOWN_COMPLEXITY` for method names (`sort`→1.5, `stream`/`collect`/`forEach`/`contains`/`indexOf`→1.0) via `method_invocation`'s named field `name` (cleaner than C++ — Java's grammar gives the method name directly, no rightmost-identifier walk needed). Separately, `object_creation_expression` nodes are checked against `COPY_CONSTRUCTOR_TYPES` (`HashSet`/`ArrayList`/`TreeSet`/etc.) — **only flagged as O(n) if the constructor actually has an argument** (`new HashSet<>(arr)` → flagged; `new HashSet<>()` → correctly NOT flagged, verified explicitly).

Verified:
```
stream_upper:   stdlib_weight=2.0   (was invisible — FIXED; 2.0 because both .stream()-equivalent and .collect() matched — see open question below)
hashset_ctor:   stdlib_weight=1.0   (was invisible — FIXED)
hashset_empty:  stdlib_weight=0.0   (correctly NOT flagged — empty constructor is O(1))
single_pass_sum: loops=1 (after enhanced_for_statement fix)
bubble_sort:    loops=2 nesting=2   (control, correct)
naive_fib:      loops=0              (control, correct)
```

**Open question worth a second look during integration**: `stream_upper` got `stdlib_weight=2.0`, not 1.0 — because both `stream` (1.0) and `collect` (1.0) matched independently in the same chain, double-counting what's really one O(n) pipeline. Decide during integration whether chained stream operations should be capped at contributing once (e.g., only weight the *first* recognized call in a method-call chain) rather than summing every recognized link. Not wrong exactly, but worth a deliberate decision rather than leaving it as an accident of the lookup table's structure.

**The honest finding**: tested the model's actual prediction on the four specific real-template-audit blind-spot cases (`std::reverse`, `std::count`, `Arrays.stream()...`, `new HashSet<>(arr)`). **All four still predict EFFICIENT**, unchanged from before. Root cause: these templates are literally filed under the `EFFICIENT` bucket in `CPP_TEMPLATES`/`JAVA_TEMPLATES` by original design, and the model is trained on stratified draws from that bucket — it learns to predict whatever the bucket is labeled, and since *other* genuinely-EFFICIENT members of the same bucket also now show nonzero `stdlib_complexity_weight` (it's not a clean discriminator within the bucket), the model has no signal telling it these four are different. **This is not a parser bug.** The parser fixed feature *extraction* — confirmed correct via direct analyzer tests and via bucket-level statistics. It cannot fix a *label* problem; no feature can. This is the same root issue as the already-documented "EFFICIENT bucket conflates O(1) and O(n) single-pass" finding, now shown to extend to "...and also conflates O(1) with O(n)-via-stdlib-call."

**What this means for next steps**: don't claim the real-template audit's specific MATCH/MISMATCH numbers improved from this work alone — they won't, because that comparison is bucket-label vs. reality, and the bucket labels didn't change. What *did* improve, verifiably: the underlying metrics (`loop_depth`, `max_nesting_depth`, `stdlib_complexity_weight`) are now real and accurate where they were previously blind. Closing the loop on the *model's prediction* requires correcting the training labels for these specific cases (or enough similar cases) — which is exactly tier 2 ("real execution") of the user's explicit "both, but parser first" decision, now with a much more precise, evidence-backed description of what tier 2 actually needs to fix.

### Python `stdlib_complexity_weight` (`PythonASTAnalyzer.visit_Call`, `profiler_main.py`)

Added a module-level `KNOWN_COMPLEXITY` dict (`sorted`: 1.5; `set`/`list`/`tuple`/`dict`/`frozenset`/`sum`/`max`/`min`/`any`/`all`: 1.0 each) and a `visit_Call` check: bare-name calls (`ast.Name`, so `set(x)` matches but `obj.set(x)` correctly doesn't) with exactly one positional argument get weighted. Two design decisions worth recording because they weren't obvious going in:

- **Deliberately not chain-capped**, unlike Java's `.stream().map().collect()` handling. `list(set(x))` weights both calls independently (2.0 total). Reasoning: Java's Stream pipeline is lazily fused by the JVM into one real pass; CPython's nested builtins have no equivalent fusion — `set()` iterates the input once, `list()` iterates the resulting set again, genuinely two passes. Capping this to 1.0 the way Java's chain is capped would have been the wrong model, not just a different convention.
- **One deliberate exception**: a bare generator-expression argument (`sum(x for x in y)`) is skipped, because that *is* a single fused pass (Python's generator protocol lazily feeds the consumer one value at a time, same fusion behavior as a Java Stream) — the loop is already counted by `visit_GeneratorExp` (§12), and weighting `sum` on top would double-count one logical pass. Verified this distinction directly: `sum(genexpr)` → `loops=1, stdlib_weight=0.0`; `sum([listcomp])` → `loops=1, stdlib_weight=1.0` (the list comprehension eagerly materializes first, so this genuinely is two passes, unlike the bare-generator case).

Wired into `profile_python()`'s `exec_time` formula with the same `*0.8` "loop-equivalent" multiplier `profile_cpp`/`profile_java` already use, applied only to the formula-estimate branch (real measured execution, when the Tier-2 sandbox is on, already inherently captures whatever `set()`/`list()` cost at runtime — doesn't need a synthetic add-on layered on top of a real number).

### C++/Java prime-sieve integer-overflow bug (`overflow_bug_report.md`) — now fixed, not just documented

The bug report already had a suggested one-line fix (bound the outer loop by `sqrt(n)`) sitting unapplied. Applying it naively would have been a *worse*, silent bug: the original code interleaves sieving and prime-collection in one loop, so bounding the whole loop by `sqrt(n)` would make the function stop *collecting primes* at `sqrt(n)` too — no crash, but silently wrong output (only primes below `sqrt(n)`, not all primes up to `n`). Caught this before it shipped by actually compiling and running the naive version's logic through by hand rather than trusting the suggested diff verbatim.

The shipped fix separates the two concerns — sieving bounded by `sqrt(n)` (via an overflow-safe `long long`/`long` cast), prime collection as a separate full pass over `2..n` — mirroring the structure the already-correct Python template used. Verified by direct compilation and execution (not just re-reading the diff): both fixed templates (`profiler_main.py`'s `CPP_TEMPLATES`/`JAVA_TEMPLATES`, kept byte-for-byte synced with `selected_templates.py`'s copies, confirmed via diff) produce the exact correct prime list at n=50 (15 primes, matches known values), correct edge cases at n=0/1/2 (0, 0, 1 primes respectively), correct count at n=100,000 (9,592, the textbook π(100,000) value), and — the actual regression check — no crash/exception at that same n=100,000, the exact input size that reproduced the original `SIGSEGV`/`ArrayIndexOutOfBoundsException`.

**Static feature effect, checked rather than assumed**: splitting one interleaved loop into two sequential ones changes `loop_depth` from 2→3 (one more top-level loop) while `max_nesting_depth` stays at 2 (the new loop isn't nested inside anything) — confirmed directly via `CodeProfiler.profile_cpp`/`profile_java` before retraining, not discovered as a surprise afterward.

**Worth being precise about, and stated plainly in both `overflow_bug_report.md` and README**: this fix does not change the real-template audit's 9/19 result. Both `cpp_prime_sieve`/`java_prime_sieve` were already `MATCH` *before* this fix, because the audit's own scaling `n` values (5,000/15,000/40,000) were deliberately chosen below the ~46,341 overflow threshold — the audit itself never exercised the crash, before or after. ### Attempt 1: real execution as a Python training feature — empirically does not discriminate as hypothesized

The plan (as scoped in §13): run each Python template through `execute_python_sandboxed()` during `generate_dataset()`, use the real measured time as a training feature instead of the formula estimate, on the theory that bubble sort's real swap-heavy cost would show up as measurably larger than a "cheap body" O(n²) neighbor's, giving the model the signal it currently lacks.

**First blocker, caught before building anything further**: `execute_python_sandboxed()` executes the code *as submitted*, and every entry in `PYTHON_TEMPLATES` is a bare function *definition* (`def f(arr): ...`), never a definition-plus-call. Verified directly: bubble sort's definition alone measures ~0.0014ms — indistinguishable from a trivial one-line function's definition-only time — because Python doesn't execute a function body until it's called. Wiring the existing sandbox in as-is would have added pure noise, not signal, to every single template.

The actual fix would require synthesizing a call with a representative argument per template. Sketched a safe version of this (only attempt single-argument templates, identified via `ast` by checking the top-level function named `f` has exactly one parameter; use a fixed synthetic list argument; critically, treat *any* exception during the synthetic call as "skip this one, fall back to the formula" rather than `execute_python_sandboxed()`'s existing "report however long it took to crash" — because for training data specifically, a fast time caused by a type-mismatch crash would be silently indistinguishable from a genuinely fast O(1) function, which would corrupt the training set in a way that's hard to detect after the fact). This design was reasoned through but deliberately not implemented, because the next test showed it wouldn't have helped anyway.

**Second blocker, the one that actually killed this approach**: sized the synthetic argument carefully first (quadruple-nested NEEDS_OPTIMIZATION-bucket templates need to stay well under the sandbox's timeout — verified n=50 gives a worst-case 187ms, comfortable margin, chose n=40 for extra headroom) — then directly tested whether *that* size actually discriminates bubble sort from a same-bucket "cheap body" O(n²) counter template. At n=40, they're statistically identical (0.055ms vs 0.073ms) — both fast enough that subprocess/interpreter overhead dominates completely over the real per-iteration cost difference. Tried larger n (300/500/1000) specifically for the O(n²) category and got a genuinely surprising result: bubble sort measured *faster* than the "cheap body" comparison at every size tested (ratio 0.76–0.86x), not slower — because a triangular loop bound (`range(len(arr) - i - 1)`, ~n²/2 total iterations) does meaningfully less total work than a fixed-bound one (`range(n)`, ~n² iterations), and CPython's per-operation costs for "modulo + conditional increment" versus "comparison + conditional swap" turned out close enough to each other that this iteration-count difference dominated, in the opposite direction from what the hypothesis needed.

The deeper, structural problem this surfaces: unlike `selected_templates.py`'s curated 19-template audit (which hand-tunes `n` per template specifically because different complexity classes need wildly different scales to produce comparably-measurable timings), a training-feature version would need to apply that same hand-tuning across all ~100+ Python templates, or accept that a single generic `n` cannot meaningfully discriminate across the full range of shapes in this dataset. That's a much larger, more uncertain undertaking than "wire up the sandbox," and — critically — even a correctly-scaled version isn't guaranteed to produce a *directional* signal favoring NEEDS_OPTIMIZATION templates, per the ratio-reversal finding above. Concluded this specific approach isn't safe to ship without a lot more validation than one session can respons­ibly provide, and did not implement it.

### Attempt 2: precisely diagnose the mechanism (not the fix) — found it, it's not a simple label bug

Compared feature-vector means at the exact bucket where the split lives (`loop_depth=2, max_nesting_depth=2`) across all three languages, not just the imbalance *ratio* §12 already found. The result is more precise than §12's: for C++ and Java, every relevant feature (`function_calls`, `conditionals`, `complexity_score`, and for C++ `memory_usage_kb` too) is *higher on average* for NEEDS_OPTIMIZATION than MODERATE at this bucket — a clean, learnable direction. For Python, every one of the same features is *lower on average* for NEEDS_OPTIMIZATION than MODERATE — backwards from what a classifier needs, which is exactly why it learns the wrong direction.

Traced this to its actual cause: Python's NEEDS_OPTIMIZATION samples at this bucket aren't only bubble sort (21 total samples, not 1 template repeated). A second template — a sum-of-factorials accumulator (`count += inner` where `inner` is a running factorial) — lands at the identical static `(loop_depth=2, max_nesting_depth=2)` point and has zero conditionals, diluting bubble sort's real conditional-having signal down to an average of 0.476. Checked whether this second template is simply mislabeled (an easy, obviously-correct fix, if so): it isn't obviously wrong — verified its real execution time grows super-linearly (n=50→0.052ms, n=100→0.248ms, n=200→0.964ms; more than the ~4x a clean O(n²) doubling would predict), consistent with genuine cost from Python's big-integer multiplication growing with the factorial's digit count as `i` increases. It's a defensible NEEDS_OPTIMIZATION label on its own real-execution merits, not a bug — meaning "just fix the mislabeled template" isn't available either; whether *this specific template* belongs at this specific static-feature bucket, generating this specific dilution, is itself the kind of labeling-philosophy call this project has consistently kept as a human decision, not a bug fix.

**Bottom line, stated plainly**: this is now a well-understood, precisely-located problem — not a mystery, not "still open" in the vague sense — but neither of the two approaches available (real execution as a feature; relabeling the diluting template) is something this session found a safe, validated way to ship. Both were tried far enough to get a real, empirical answer rather than a guess, and both came back "doesn't cleanly work" rather than "works." Recording the precise mechanism here so whoever picks this up next starts from "here's exactly why, and here's what's already been ruled out," not from zero.

## 20. "No gaps" — built the argument-shape detector after all, found a second mislabel, closed 5 of 7 mismatches

Prompted with "wrap up everything, I want no gaps." §19 had explicitly declined to build a recursion-shape feature, reasoning that a naive self-call-count version would misclassify merge sort/quicksort/tree traversal. Given the explicit instruction, attempted a *conservative* version instead of declining again — one that only flags recursion when confident it isn't divide-and-conquer, tested against every existing template before being trusted rather than assumed safe. ### `recursive_branching_risk` — built for all three languages, validated before shipping

Design: for a self-recursive function with 2+ self-calls, check whether *any* call's argument matches one of three "shrinking" patterns — division by a constant ≥2 (directly or through a tracked local variable), a filtered comprehension (directly or through a tracked variable), or field/attribute access combined with an explicit null/None base-case check. If none match, flag with risk = self-call count.

**The first prototype had a real bug, caught by testing against the full template corpus before trusting it, not after shipping**: quicksort (`f(l)+f(r)` where `l`/`r` are assigned via `[x for x in arr if ...]` *earlier* in the function, not at the call site) was flagged as dangerous, because the first version only checked the argument's shape *at the call site* — it didn't trace back through variable assignments the way the division-tracking already did. Fixed by tracking "filter variables" the same way as "shrink variables," then re-tested against all four hand-picked cases (naive Fibonacci, merge sort, quicksort, tree traversal) plus a full sweep of every template in `PYTHON_TEMPLATES`: zero false positives across all 82 EFFICIENT/MODERATE templates, exactly the genuinely-dangerous ones flagged in NEEDS_OPTIMIZATION.

**A second, unrelated mislabel surfaced from that same sweep, not gone looking for**: a naive-Fibonacci variant differing from the already-correct one only in its base case (`n<=2: return 1` vs `n<=1: return n`) was filed `MODERATE` in Python *and* Java — same algorithm, same exponential growth (verified: 109 → 13,529 → 1,664,079 calls at n=10/20/30), inconsistently labeled from its sibling for no apparent reason beyond however the templates were originally authored. Fixed in both languages.

**A real editing mistake happened while fixing it, caught before moving on**: the first `str_replace` on the Python template list, needing 6 lines of context to be unique, replaced all 6 with only 1 — silently deleting four unrelated, correctly-labeled templates (Sudoku solver, N-Queens, a permutation-backtracking variant, subset-sum DFS) that had nothing to do with the fix. Caught by checking the bucket's template count against the expected number (23, not 18) before continuing, not after. All four restored in original order; final counts checked programmatically, not eyeballed. The C++ instance of the same fix was on an isolated line and had no such risk.

**Ported to C++ and Java via tree-sitter**, using each language's actual field names rather than assumed ones — checked directly via the tree-sitter Python bindings before writing any detection logic (`function_definition`→`declarator`→`declarator` for C++'s function name, `method_declaration`'s direct `name` field for Java, `field_expression` vs `field_access` for member/attribute access, `null` vs `null_literal` for the two languages' respective null checks). C++ has exactly one 2-self-call template in its whole corpus (a merge-sort-style index-based divide, using division through a local variable — same pattern as Python's, verified not to false-positive); Java has zero divide-and-conquer templates with 2+ self-calls at all, so had no false-positive risk to check beyond the sanity cases. Both validated with a full-corpus sweep before being trusted, same as Python.

**Retrained and re-ran the full 230-template audit**: naive Fibonacci fixed in *both* Python and Java (the two languages that had it) — the feature did exactly what it was built for, zero new mismatches anywhere. **5/230 → 3/230.**
